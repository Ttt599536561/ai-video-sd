# 文档维护规则

## 设计原则

本项目文档参考 Anthropic 对 Claude Code 项目记忆的建议：短入口、分层链接、把高频上下文放在容易被助手读取的位置。根目录的 `CLAUDE.md` 只保留每次会话都需要的事实，避免把实现细节和历史过程塞进去。

## 分层规则

- `CLAUDE.md`：给 AI 助手看的短记忆，只放先读顺序、项目一句话和硬约束。
- `PROJECT.md`：给人和 AI 看的全局文档，写背景、目标、当前状态和文档地图。
- `docs/*/README.md`：目录级索引，只负责解释本目录有什么。
- 详细模块文档：放具体接口、业务规则、数据流和待办。
- 上下文文档：只记录恢复任务需要的信息，不复述所有历史。
- `docs/context/new-chat-prompt.md`：新开对话时复制给助手的极简接力提示词，只保留继续开发需要的事实和下一步。

## 上下文瘦身规则

- 当前窗口上下文保留“正在做什么、已完成什么、下一步做什么、阻塞点”。
- 完成事项先用删除线保留一小段时间，再合并到长期上下文或删除。
- `current-session.md` 建议控制在约 50 行内；如果完成项开始像流水账，应合并成里程碑。
- 长期上下文只沉淀稳定决策、架构事实和长期约束。
- 不在上下文文档里粘贴大段代码、完整接口返回、完整日志。
- 每次大任务结束时，优先更新 `docs/context/current-session.md`。
- 如果新对话反复消耗大量上下文，优先更新 `docs/context/new-chat-prompt.md`，不要把长聊天记录复制到新窗口。

## 何时新增文档

- 一个模块开始有独立业务规则时，新增到 `docs/modules/`。
- 一个需求会影响多个模块时，新增到 `docs/requirements/`。
- 一个部署或排障步骤需要重复执行时，新增到 `docs/operations/`。
- 单次临时探索不新增文档，除非结论会影响后续决策。
